## 1. 計畫概覽

### 1.1 技術架構
- **PL層（展示層）：** Flutter (APP) + LINE OA
- **APL層（應用邏輯層）：** RESTful API服務
- **BL層（商業邏輯層）：** Node.js模組
- **DL層（資料層）：** Firebase Authentication + Firestore + Cloud Storage

### 1.2 開發策略
採用敏捷開發方式，以7個Phase漸進式交付，每個Phase專注於特定功能領域，確保用戶價值最大化與技術風險控制。

---

## 2. Sprint架構說明

### 2.1 四層架構對應
每個Phase均包含完整的四層實作：
- **PL層：** 用戶介面與互動體驗
- **APL層：** API規格設計與實作
- **BL層：** 核心商業邏輯模組
- **DL層：** 資料結構與儲存配置

---

## 3. Phase 1: 核心進入流程 (雙軌並行策略)

### 3.1 Phase目標
建立使用者可以成功註冊、登入、進行基本記帳的完整流程，確保系統核心價值鏈可運行。

### 3.2 雙軌並行策略說明

基於技術風險分散與開發效率最大化考量，Phase 1採用雙軌並行開發模式：

#### **P1-1: APP UI到DL的完整垂直切片**
**目標：** 用戶端完整體驗實作
**範圍：** Flutter APP UI → 本地狀態管理 → Mock API → 直接Firebase整合

#### **P1-2: PL到DL的服務端整合** 
**目標：** 服務端四層架構完整串接
**範圍：** PL函數邏輯 → APL API服務 → BL商業邏輯 → DL資料服務

### 3.3 P1-1功能範圍（APP垂直切片）

#### 3.3.1 PL層（Flutter APP完整實作）
- **S-102:** APP 啟動頁
- **S-103:** APP 註冊頁（含四模式評估）
- **S-104:** APP 登入頁
- **S-105:** 忘記密碼頁
- **S-106:** 模式評估問卷頁
- **S-107:** 歡迎完成頁
- **S-108:** 主頁（四模式差異化首頁）
- **S-202:** APP 記帳主頁
- **S-203:** APP 記帳表單頁（核心記帳功能）

#### 3.3.2 本地服務層（Mock API實作）
- 本地認證狀態管理
- 本地記帳資料快取
- Mock API服務（模擬8101、8102、8103端點）

#### 3.3.3 DL層（直接Firebase整合）
- Firebase Authentication直接整合
- Firestore基本操作（不經過BL層）
- 本地資料同步機制

### 3.4 P1-2功能範圍（服務端整合）

#### 3.4.1 PL層（函數邏輯實作）
- **7301.系統進入功能群.dart**：核心函數邏輯
- **7302.記帳核心功能群.dart**：記帳函數邏輯
- PL→APL呼叫介面

#### 3.4.2 APL層（API服務完整實作）
- **8101:** 認證服務 API（註冊、登入、模式評估、密碼重設）
- **8102:** 用戶管理服務 API（個人資料、模式切換、偏好設定）
- **8103:** 記帳交易服務 API（基本記帳CRUD、統計查詢）
- **8301、8302、8303:** 對應路由代碼實作

#### 3.4.3 BL層（商業邏輯層完整實作）
- **1320. WH.js:** Webhook模組（LINE OA記帳）
- **1309. AM.js:** 帳號管理模組（認證、模式管理）
- **1301. BK.js:** 記帳核心模組（基本記帳功能）
- **1331. DD1.js:** 核心協調模組（科目管理）
- **1311. FS.js:** Firestore操作模組優化

#### 3.4.4 DL層（資料層完整配置）
- Firebase Authentication設定
- Firestore完整資料結構：users、ledgers、transactions集合
- 四層架構資料流優化

### 3.5 執行時程安排

#### **Week 1-3: P1-1執行（APP垂直切片）**
- Flutter APP完整打包與部署
- Mock API服務建立
- APP基本功能完整測試
- 用戶體驗優化

#### **Week 2-4: P1-2執行（服務端整合）** 
- PL→APL→BL→DL四層整合
- RESTful API服務部署
- SIT測試執行（TC-SIT-008~028）
- 服務穩定性驗證

#### **Week 5: P1整合驗證**
- P1-1 APP連接P1-2真實API服務
- 端到端整合測試
- 完整用戶旅程驗證
- 效能與穩定性測試

### 3.6 測試策略調整

#### **P1-1測試重點**
- **UI/UX測試：** 所有APP畫面功能驗證
- **Mock API測試：** 模擬服務回應測試
- **APP性能測試：** 啟動速度、記憶體使用、電池消耗
- **本地資料測試：** 快取機制、資料同步

#### **P1-2測試重點**
- **API整合測試：** TC-SIT-001~007重點執行
- **四層架構測試：** PL→APL→BL→DL完整驗證
- **商業邏輯測試：** BL層函數單元測試
- **資料一致性測試：** Firestore操作驗證

#### **P1整合測試**
- **端到端測試：** APP + API完整串接
- **用戶旅程測試：** 註冊→登入→記帳完整流程
- **負載測試：** API服務壓力測試
- **回歸測試：** 確保整合後功能正常

### 3.7 LINE OA功能（獨立開發）

#### 3.7.1 PL層
- **S-101:** LINE OA 歡迎頁（統一四模式入口）
- **S-201:** LINE OA 記帳對話（基本文字記帳）

#### 3.7.2 整合點
- LINE OA可獨立使用P1-2的API服務
- 與APP共享相同的後端服務架構
